# 研究生招生信息管理系统 — 项目设计说明书

---

## 一、项目背景与目标

### 1.1 项目概述

研究生招生信息管理系统旨在实现高校研究生招生工作的全面信息化管理。系统覆盖招生全流程的三个核心阶段：

- **第一阶段 — 考生报名**：考生档案录入与管理，报名信息汇总
- **第二阶段 — 初试与复试**：初试成绩录入、分数线筛选、复试成绩录入、录取判定
- **第三阶段 — 录取分析统计**：成绩统计分析、录取生源分析、计划与实际招生对比

### 1.2 建设目标

1. 替代手工 Excel 管理模式，实现数据集中存储
2. 提供 RESTful API 接口，支持前后端分离
3. 实现自动化的成绩汇总计算与统计分析
4. 为招生决策提供数据支撑（各科及格率、分数段分布、生源分析）

## 三、功能模块设计

### 3.1 模块总览

| 模块编号 | 模块名称 | 对应阶段 | 功能描述 |
|----------|----------|----------|----------|
| M1 | 专业字典管理 | 全局 | 专业代码、名称、计划招生数的增删改查 |
| M2 | 考生档案管理 | 第一阶段 | 考生报名信息录入、修改、查询、删除 |
| M3 | 初试成绩管理 | 第二阶段 | 初试三科成绩录入、总分自动计算 |
| M4 | 复试成绩管理 | 第二阶段 | 复试三环节成绩录入、总分自动计算 |
| M5 | 录取名单管理 | 第二阶段 | 录取/取消录取、综合总分记录 |
| M6 | 统计分析 | 第三阶段 | 成绩分析 + 录取生源分析 |

### 3.2 模块详细功能

#### M1 专业字典管理

| 功能 | API | 方法 | 说明 |
|------|-----|------|------|
| 专业列表 | `/api/major/list` | GET | 返回所有专业 |
| 按代码查询 | `/api/major/{majorCode}` | GET | 查询单个专业 |
| 新增专业 | `/api/major/add` | POST | 新增专业信息 |
| 更新专业 | `/api/major/update` | PUT | 修改专业信息 |
| 删除专业 | `/api/major/delete/{majorCode}` | DELETE | 删除专业 |

#### M2 考生档案管理

| 功能 | API | 方法 | 说明 |
|------|-----|------|------|
| 考生列表 | `/api/candidate/list` | GET | 返回所有考生 |
| 按考号查询 | `/api/candidate/{examId}` | GET | 查询单个考生 |
| 按专业查询 | `/api/candidate/major/{majorCode}` | GET | 按报考专业筛选 |

| 按考号查询 | `/api/first-test/{examId}` | GET | 查询单个考生初试成绩 |
| 录入成绩 | `/api/first-test/add` | POST | 录入政治、外语、专业基础 |
| 修改成绩 | `/api/first-test/update` | PUT | 更新初试成绩 |
| 删除成绩 | `/api/first-test/delete/{examId}` | DELETE | 删除初试成绩 |

#### M4 复试成绩管理

| 功能 | API | 方法 | 说明 |
|------|-----|------|------|
| 成绩列表 | `/api/second-test/list` | GET | 返回所有复试成绩（含自动计算的总分） |
| 按考号查询 | `/api/second-test/{examId}` | GET | 查询单个考生复试成绩 |
| 录入成绩 | `/api/second-test/add` | POST | 录入专业科目、面试、上机 |
| 修改成绩 | `/api/second-test/update` | PUT | 更新复试成绩 |
| 删除成绩 | `/api/second-test/delete/{examId}` | DELETE | 删除复试成绩 |

#### M5 录取名单管理

| 功能 | API | 方法 | 说明 |
|------|-----|------|------|
| 录取列表 | `/api/admission/list` | GET | 查询所有已录取考生 |
| 按考号查询 | `/api/admission/{examId}` | GET | 查询考生录取状态 |
| 录取考生 | `/api/admission/admit` | POST | 将考生加入录取名单 |
| 取消录取 | `/api/admission/cancel/{examId}` | DELETE | 从录取名单中移除 |

#### M6 统计分析

| 功能 | API | 方法 | 说明 |
|------|-----|------|------|

## 六、核心业务流程

### 6.1 招生完整流程

```
考生报名               初试              复试              录取
  │                    │                │                 │
  ▼                    ▼                ▼                 ▼
┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────────┐
│录入考│→│录入初│→│按分数│→│录入复│→│计算综│→│ 加入录取  │
│生档案│  │试成绩│  │线筛选│  │试成绩│  │合总分│  │ 名单公布  │
└──────┘  └──────┘  └──────┘  └──────┘  └──────┘  └──────────┘
                                                         │
                                                         ▼
                                                  ┌──────────────┐
                                                  │ 统计分析     │
                                                  │ ·成绩分析    │
                                                  │ ·生源分析    │
                                                  │ ·计划vs实际  │
                                                  └──────────────┘
```

### 6.2 成绩计算逻辑

- **初试总分** = 政治成绩 + 外语成绩 + 专业基础科目成绩（数据库虚拟列自动计算）
- **复试总分** = 复试专业科目 + 面试成绩 + 上机成绩（数据库虚拟列自动计算）
- **综合总分** = 最终初试成绩 + 最终复试成绩（录取时应用层计算并记录）

### 6.3 分数线筛选逻辑（待实现）

```
if (政治 >= 单科线 AND 外语 >= 单科线 AND 专业基础 >= 单科线 AND 初试总分 >= 总分线):
    → 进入复试
else:
    → 不通过
```

## 七、统一返回格式

系统定义了 `Result<T>` 统一返回格式，所有 API 接口均以此格式返回：

```json
{
    "code": 200,
    "message": "操作成功",
    "data": { ... }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | Integer | 200 = 成功，500 = 失败 |
| `message` | String | 操作结果描述 |
| `data` | T（泛型） | 返回的业务数据，可为对象、数组或 null |

## 八、安全性设计

### 8.1 已实现

- **CORS 跨域配置**：后端配置了 CorsConfig，允许前端开发服务器（localhost:5173）跨域访问
- **事务管理**：Service 层关键写操作使用 `@Transactional` 注解确保数据一致性
- **前端代理**：Vite 开发服务器配置了 `/api` 代理到后端 8080 端口，避免直接暴露后端

### 8.2 待实现的加分项

| 安全措施 | 优先级 | 说明 |
|----------|--------|------|
| HTTPS | 中 | 生产环境配置 SSL 证书 |
| 敏感数据加密 | 高 | 对考生身份证号、联系方式等加密存储 |
| Redis 缓存 | 中 | 缓存专业字典等高频读取数据 |
| 分布式锁 | 低 | 使用 Redisson 防止并发修改 |
| 多线程统计 | 低 | 异步执行统计任务，提升性能 |

## 九、部署方案

### 9.1 开发环境部署

```bash
# 1. 初始化数据库
mysql -u root -p < database/schema.sql

# 2. 启动后端
cd backend
mvn spring-boot:run
# 后端启动在 http://localhost:8080

# 3. 启动前端
cd frontend
npm install
npm run dev
# 前端启动在 http://localhost:5173
```

### 9.2 生产环境部署

```bash
# 后端打包
cd backend
mvn clean package -DskipTests
java -jar target/admission-0.0.1-SNAPSHOT.jar

# 前端打包
cd frontend
npm run build
# 将 dist/ 目录部署到 Nginx
```

### 9.3 Nginx 配置示例

```nginx
server {
    listen 80;
    server_name your-domain.com;

    # 前端静态文件
    location / {
        root /var/www/admission-frontend/dist;
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 代理
    location /api/ {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```
